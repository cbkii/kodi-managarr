# Managarr agent instructions

These instructions apply to the entire repository. Jules, Codex, GitHub Copilot coding agent, Copilot code review, and any other autonomous coding agent must follow them. More-specific `AGENTS.md` files may refine these rules for a subtree, but may not weaken the safety, validation, or source-verification requirements here.

## Mission

Architect, engineer, review, test, and maintain `context.arr.manager`, a Kodi Python context-menu add-on that lets a Kodi user manage Radarr movies and Sonarr series/episodes from an Android TV.

The add-on must remain:

- safe for destructive media operations;
- reliable on Kodi 19+ with Python 3, especially Kodi on Android TV;
- compatible with Debian-hosted Radarr and Sonarr on an ARM64 Raspberry Pi;
- conservative when Kodi metadata, Servarr history, paths, or API responses are ambiguous;
- installable directly from the release ZIP generated by this repository;
- testable outside Kodi using standard Python tooling.

Prowlarr is not currently a deletion backend. Treat it as an optional future indexer-management integration and do not route movie, series, episode, file, history, or blocklist operations through it.

## Source hierarchy

Use sources in this order:

1. The target application's live OpenAPI/Swagger description for the deployed version.
2. The matching release/tag or current upstream source code for Kodi, Radarr, Sonarr, or Prowlarr.
3. Official Kodi and Servarr documentation.
4. Mature, actively maintained Kodi add-ons for implementation patterns.
5. Community guidance only as supporting context, never as the sole authority for an API or destructive operation.

If sources conflict, prefer the source matching the deployed version and document the decision in code comments, tests, or the PR description. Never blindly copy another add-on: confirm that its Kodi generation, Python API, licence, and execution model match this project.

- `addon.xml`: add-on identity, dependencies, context-menu registration, metadata, version and news.
- `context.py`: Kodi context-item entry point.
- `default.py`: executable/settings and diagnostics entry point.
- `resources/settings.xml`: user-facing settings schema.
- `resources/language/`: localised strings. New user-visible text must be localised.
- `resources/lib/arr_manager/actions.py`: destructive workflows and orchestration.
- `resources/lib/arr_manager/clients.py`: Radarr and Sonarr API adapters.
- `resources/lib/arr_manager/http.py`: HTTP transport, authentication, URL construction and error handling.
- `resources/lib/arr_manager/resolver.py`: Kodi-to-Servarr entity resolution.
- `resources/lib/arr_manager/history.py`: imported-release/history matching.
- `resources/lib/arr_manager/fileops.py`: Kodi VFS and optional SFTP file operations.
- `resources/lib/arr_manager/config.py`: settings parsing, path mapping and validation.

Keep responsibilities separated. API resources and transport belong in clients/HTTP, matching in resolver/history, file operations in fileops, Kodi-specific UI in kodi/entrypoints, and workflow ordering in actions.

## Non-negotiable architecture and safety invariants

### 1. Servarr API is the primary authority

Prefer Radarr/Sonarr API operations so the application updates its database and filesystem coherently. Never edit Servarr SQLite databases, `config.xml`, queue data, history, blocklists, or application state files directly.

Use:

- Radarr API v3 unless the target instance's OpenAPI states otherwise.
- Sonarr API v3 unless the target instance's OpenAPI states otherwise.
- Prowlarr API v1 unless the target instance's OpenAPI states otherwise.

Do not assume all Servarr applications expose the same API version, resource names, query parameters, bulk semantics, or command payloads.

### 2. Destructive actions must fail closed

Before deleting anything:

- resolve exactly one intended entity/file;
- validate the media type;
- validate required API configuration and backend configuration;
- preflight any history/blocklist evidence needed by the action;
- reject empty, root, share-root, parent, protected, malformed, or unmapped paths;
- show a specific confirmation unless confirmation is explicitly disabled;
- honour dry-run mode without mutating any remote or local state.

Never replace an ambiguity with "first result", fuzzy best guess, unchecked index, or implicit fallback.

### 3. Resolution order is stable identifiers first

For movies:

1. TMDb ID;
2. mapped canonical file/path evidence;
3. exact normalised title and year.

For series/episodes:

1. TVDb ID;
2. mapped canonical file/path evidence;
3. exact normalised title and year;
4. season/episode and shared episode-file relationships where applicable.

Reject multiple matches or weak matches. Preserve multi-episode-file handling: deleting one physical episode file may affect several Sonarr episode records.

### 4. Delete & Exclude semantics

- Movie: delete the Radarr movie/files/folder and create the Radarr import exclusion.
- Series: delete the Sonarr series/files/folder and create the Sonarr import-list exclusion.
- Episode: Sonarr has no equivalent episode-level import-list exclusion. Unmonitor every episode linked to the selected physical file, then delete that file. Restore monitoring state if deletion fails part-way through.

Do not describe episode unmonitoring as a native Sonarr exclusion. Keep the UI and documentation explicit.

### 5. Delete & Replace semantics

A replacement workflow must:

1. resolve the exact media entity and physical file;
2. identify the corresponding imported-release history record before deletion;
3. stop without deletion when strict blocklist matching is enabled and evidence is insufficient;
4. delete the file through the selected backend;
5. use Servarr's normal failed-history/blocklist path for the matched import;
6. start the exact movie, episode-set, or series search;
7. avoid searching when a prerequisite deletion/blocklist operation failed;
8. present errors that state which stage failed.

Imported history uses `DownloadFolderImported` (`eventType=3`) in the supported Radarr/Sonarr generations. Before changing this assumption, verify upstream enums/controllers and add fixtures for the new response shape.

The failed-history endpoint is intended to invoke Servarr's normal failed-download handling. Do not invent a local blocklist format or write directly to application data.

### 6. Direct file backends require reconciliation

When Kodi VFS or SFTP deletes a file directly:

- issue the appropriate Radarr/Sonarr rescan command;
- poll the relevant API resource until the deleted file record disappears or a bounded timeout expires;
- do not rely on an arbitrary fixed sleep;
- do not start replacement search until reconciliation succeeds;
- close SFTP resources in `finally` paths.

Servarr commands are asynchronous. When later behaviour depends on command completion, retain the returned command ID and poll `/command/{id}` or the version-specific equivalent until terminal success/failure with a bounded timeout.

### 7. Filesystem boundary rules

Kodi on Android is not normal desktop Linux.

- Use `xbmcvfs` for `smb://`, `special://`, and other Kodi VFS paths.
- Do not pass Kodi VFS URLs to `os`, `pathlib`, `shutil`, or shell commands.
- Use `xbmcvfs.translatePath` only where a real local path is required and supported.
- Use Kodi `special://profile`, `special://home`, or `special://temp` locations, never assume `/tmp` exists or is appropriate on Android.
- Do not assume Android Kodi provides a system `ssh` executable, writable arbitrary filesystem locations, desktop keyrings, or binary Python wheels.
- Keep SSH/SFTP optional and dependency-detected. Prefer the API or Kodi VFS backend.
- Never recursively delete a configured media root, SMB share root, filesystem root, or an ancestor of a protected path.
- Preserve spaces, Unicode, URL escaping, SMB credentials handled by Kodi, and case-sensitive Debian paths.

### 8. Kodi integration rules

- Maintain `kodi.context.item` registration and visibility expressions in `addon.xml`.
- Kodi 19+ context-item `args` arrive via `sys.argv`.
- The selected context item is available as `sys.listitem`; prefer its modern `InfoTagVideo` data and retain tested fallbacks for incomplete skins/builds.
- Avoid global Kodi imports in pure modules so host-side unit tests remain runnable.
- Keep UI calls and notifications in the Kodi adapter/entry points.
- Use Kodi logging, but never log API keys, passwords, embedded SMB credentials, private-key material, or complete sensitive URLs.
- Keep network operations bounded by explicit connect/read timeouts. Context-menu actions must not hang the Kodi UI indefinitely.
- Make cancellation and progress behaviour explicit for multi-file operations.
- Settings marked hidden are only visually obscured; Kodi stores them unencrypted. Treat them as secrets in logs and diagnostics.
- Add all visible strings to the language catalogue rather than hard-coding English in runtime UI.

### 9. HTTP and API client rules

- Construct URLs safely when a user configures a base URL with a reverse-proxy path.
- Authenticate using the application-supported API-key mechanism, normally `X-Api-Key`; verify against current OpenAPI/source before adding alternatives.
- Preserve TLS verification by default. An opt-out may exist for private self-signed deployments but must remain explicit.
- Encode query parameters rather than concatenating raw strings.
- Distinguish authentication, permission, not-found, validation, rate-limit, connection, timeout, TLS, malformed JSON, and server failures where practical.
- Include safe response context in errors, never secret headers or credentials.
- Validate response shapes before indexing fields used for destructive decisions.
- Do not introduce retries for non-idempotent deletion or command requests unless idempotency and duplicate effects are proven.
- If adding Prowlarr, create a separate client/version boundary. Prowlarr manages indexers and applications; it is not interchangeable with Sonarr/Radarr media resources.

### 10. Debian/Servarr deployment assumptions

The normal deployment is Debian on ARM64, but code must communicate over HTTP and must not depend on one installation layout.

Official manual installation guidance commonly uses:

- binaries under `/opt/Radarr`, `/opt/Sonarr`, `/opt/Prowlarr`;
- app data under `/var/lib/radarr`, `/var/lib/sonarr`, `/var/lib/prowlarr`;
- dedicated service users and systemd units;
- a shared media group for applications that must read/write media.

Treat those as operational guidance, not hard-coded application paths. Never solve an API/add-on issue by broad `chmod 777`, running Servarr as root, changing systemd users, or mutating Debian permissions from Kodi. Explain required least-privilege ownership/group changes separately when deployment work is requested.

## Testing requirements

Always run from repository root:

```bash
python3 scripts/validate.py
python3 -m unittest discover -s tests -v
python3 scripts/package.py
```

Then verify the ZIP:

```bash
python3 - <<'PY'
import xml.etree.ElementTree as ET
import zipfile
from pathlib import Path

root = ET.parse('addon.xml').getroot()
addon_id = root.attrib['id']
version = root.attrib['version']
archive_path = Path('dist') / f'{addon_id}-{version}.zip'
assert archive_path.is_file() and archive_path.stat().st_size > 0
with zipfile.ZipFile(archive_path) as archive:
    names = archive.namelist()
    assert names and all(name.startswith(addon_id + '/') for name in names)
    assert f'{addon_id}/addon.xml' in names
    assert archive.testzip() is None
print(archive_path)
PY
```

Delete generated `dist/` artefacts before committing unless the task explicitly requires committed build outputs.

Tests for changed behaviour must cover applicable cases from this matrix:

- successful Radarr movie and Sonarr series/episode flows;
- multi-episode files;
- no match, duplicate match and weak history match;
- strict blocklist mode and non-strict mode;
- dry run and user cancellation;
- API backend, Kodi VFS backend and optional SFTP adapter boundaries;
- protected/root/share-root path rejection;
- mapped/unmapped paths, spaces, Unicode and trailing separators;
- reverse-proxy base paths and query encoding;
- 401/403/404/409/422/429/5xx, timeout, TLS and malformed-response handling;
- partial failure and rollback/restoration behaviour;
- command polling success, failure and timeout when command completion matters;
- secret redaction in logs and diagnostics;
- package contents and version consistency.

Use fakes/mocks for Kodi and Servarr. Unit tests must not require a real Kodi installation, network, Radarr, Sonarr, Prowlarr, SMB server, SFTP server, or destructive filesystem access. Add an opt-in integration harness only when credentials/URLs are supplied through environment variables and destructive operations are impossible or explicitly sandboxed.

## Android/Kodi device verification

Host tests are necessary but not sufficient for release-critical Kodi changes. Where possible, document manual verification on Kodi for Android:

- ZIP installs without extraction or dependency errors;
- context submenu appears only for supported library item types;
- settings open, save and reload;
- selected metadata resolves correctly for movie, series and episode items;
- confirmation, cancellation, dry run and notifications render correctly with a TV remote;
- API requests work over the Android TV's LAN route;
- Kodi SMB credentials and VFS deletion work without exposing credentials;
- errors return control to Kodi instead of leaving a spinner/hung script;
- library refresh occurs after successful changes;
- upgrade from the previous released ZIP preserves settings.

If device testing is unavailable, say so plainly and identify exactly what remains unverified.

## Review standard

When reviewing a PR or branch:

- read the complete diff and every unresolved review thread;
- inspect collapsed logs and full CI output rather than relying on summaries;
- verify API names, payloads and query parameters against current upstream sources;
- trace destructive workflows end-to-end, including failure ordering;
- check that direct deletion cannot escape configured media paths;
- check that a search cannot run after an unconfirmed blocklist/deletion failure;
- look for secrets in logs, fixtures, settings defaults and diagnostics;
- confirm tests assert behaviour rather than implementation trivia;
- run validation/tests/package checks and inspect the produced ZIP;
- classify comments as valid, stale, duplicate, already fixed, out of scope, or needing human input;
- resolve review threads only when the issue is genuinely complete.

Do not approve changes that merely make CI green while weakening fail-closed behaviour, history matching, path protection, TLS defaults, secret handling, or Android compatibility.

## Release rules

`.github/workflows/release.yml` is the authoritative release path.

- Releases are manual `workflow_dispatch` runs.
- Dispatch fields control branch, exact version, channel, release notes and Latest status.
- A blank version increments the highest numeric `x.y.z` patch version.
- The workflow writes the selected version and news to `addon.xml`, validates, tests, packages, commits metadata, tags that commit and publishes the ZIP plus SHA-256 file.
- Do not manually edit a tag, publish a duplicate release, or commit generated ZIPs.
- Any release-workflow change must preserve Kodi-valid ZIP root layout: `context.arr.manager/...`.
